# Panda 需求文档

## 1. 目标

构建一个面向移动端优先的本地 AI 编码代理远程控制系统。

当前目标：

- Codex

规划目标：

- Claude Code

## 2. 产品目标

1. 能够在手机上远程查看和控制 AI 编码会话。
2. 管理多台机器，而不是只管理一台。
3. 自动发现那些在 Panda 之外启动的现有本地会话。
4. 在同一个 UI 中提供 git、终端和项目预览操作。
5. 在架构上保持对 provider 中立。
6. 默认采用 Tailscale，而不是引入公网 relay 的复杂性。

## 3. MVP 的非目标

- 原生 iOS/Android 应用
- 自行运营的公网 relay
- 企业级多用户 RBAC
- 第一天就实现与 Claude Code 的完整对等能力
- 对每一种外部会话类型都实现完美的审批接管

## 4. 功能需求

## 4.1 远程控制

系统必须：

- 通过 Tailscale 工作
- 不需要公网 IP
- 允许手机访问活动中的会话
- 显示实时输出更新
- 允许提交 prompt
- 在 provider 能力存在时允许中断
- 在 provider 能力存在时允许执行审批操作

## 4.2 多 Agent / 多机器

系统必须：

- 显示所有在线的 agent 机器
- 允许选择其中一台机器
- 显示机器元数据：
  - 名称
  - provider 可用性
  - 在线/离线
  - 当前项目/会话数量
- 允许在选中的机器上启动新会话

更理想的是：

- 有一个可选的管理节点来聚合所有 agent
- 即使没有那个管理节点，agent 仍然可用

## 4.3 会话管理

系统必须：

- 自动发现现有的 Codex 会话
- 按机器和项目分组显示会话
- 支持新建项目
- 支持新建会话
- 显示会话的实时更新
- 显示历史会话
- 为每个会话标注能力等级

必须使用的会话标签：

- `Managed`
- `Attached Live`
- `History Only`

## 4.4 现有会话互操作

系统不应要求用户必须通过 `panda codex` 启动每一个会话。

系统必须尝试附着到：

- 普通 `codex`
- Codex 桌面应用会话

后续：

- 普通 `claude`
- Claude Code 的日志/hooks

MVP 验收标准：

- 外部创建的 Codex 会话会自动出现
- 这些会话的内容能够接近实时地刷新
- Panda 受管会话拥有完整控制能力

## 4.5 Git 管理

系统必须：

- 自动读取 git status
- 显示变更文件
- 显示 diff
- 回滚变更
- stage 和 unstage
- commit
- fetch/pull/push
- 创建分支
- 切换分支
- 创建 worktree
- 切换 worktree

## 4.6 终端与运行时

系统必须：

- 按项目启动已保存的运行/调试命令
- 实时流式传输终端输出
- 显示通过 Panda 运行时管理器创建的运行中进程
- 检测预览 URL/端口
- 让手机能够通过 tailnet 打开这些预览 URL

## 4.7 语音输入

系统应当：

- 在 Web 客户端中支持语音转文本输入
- 将转写后的文本插入当前活动会话的输入框

## 5. UI 需求

## 5.1 视觉方向

UI 应当：

- 在界面密度上接近 Codex 桌面应用
- 在移动端触控人体工学上接近 Claude Code
- 以白色作为主要强调色系统
- 避免通用的紫色加白底 AI 风格

## 5.2 主要界面

必需界面：

- 机群仪表盘
- 机器详情
- 项目列表
- 会话列表
- 会话时间线
- git 面板
- 终端面板
- 预览面板
- 设置 / 诊断

## 5.3 会话界面

会话视图必须显示：

- 消息流
- 在可用时显示思考/工具活动
- 审批请求
- 变更文件
- 当前分支/worktree
- 终端日志
- 预览 URL

## 6. 技术需求

## 6.1 网络

- 以 Tailscale 为优先
- tailnet 上的 HTTPS 或安全 WebSocket
- MVP 中不强制要求公网 relay

## 6.2 架构

- 与 provider 无关的适配器层
- 可选的中心 Hub
- 每台机器一个 agent 守护进程
- 实时事件流
- 具备能力感知的会话模型

## 6.3 存储

Agent 本地存储：

- 轻量级本地元数据
- 缓存的会话索引
- 预览/运行时状态

Hub 存储：

- 可选
- 机器注册表
- 机群会话缓存
- 通知状态

## 7. MVP 范围

## 第一阶段

- 仅支持 Codex
- 仅支持 Tailscale
- Panda Agent
- 最小化 Panda Hub 或直接 agent 模式
- 移动端 PWA
- 自动发现 Codex 会话
- 对现有 Codex 会话进行实时镜像
- 对 Panda 受管 Codex 会话提供完整控制
- git 面板
- 终端面板
- 预览面板
- 基础语音输入

## 第二阶段

- 更强的多机器管理界面
- 推送通知
- 更好的审批 UX
- 会话搜索
- 项目模板
- 更好的外部会话附着质量

## 第三阶段

- Claude Code 支持
- 更丰富的多 agent 编排
- 可选 relay 模式
- 如果需要则提供原生移动端包装

## 8. 验收标准

## MVP 验收清单

- 手机可以通过 Tailscale 打开 Panda，并看到所有在线 agent。
- 用户可以选择一台机器并看到它的项目和会话。
- 在 Panda 外部启动的现有 Codex 会话会自动出现。
- 至少有一个外部启动的 Codex 会话可以在手机上实时刷新。
- 用户可以在手机上启动一个新的受管 Codex 会话。
- 受管会话支持发送 prompt、实时流和审批流程。
- 活动项目的 git status 和 diff 可见。
- 用户可以执行回滚、提交、拉取、推送、切换分支以及创建 worktree。
- 用户可以运行一个已保存的开发命令，并远程查看日志。
- 用户可以通过 Tailscale 在手机上打开一个预览 URL。

## 9. 需要保持坚定的产品决策

1. 不要强制只允许通过 wrapper 启动。
2. MVP 不要强制引入公网 relay。
3. 不要在核心模型里把只支持 Codex 的假设写死。
4. 不要假装每个会话都可以被完全控制。
5. 不要交付一个没有 git/terminal/preview 的纯聊天外壳。

## 10. 最终需求总结

Panda 应当是：

- 面向移动端优先
- 以 Tailscale 为优先
- 面向多机器
- 以会话互操作为优先
- 感知 git
- 感知 terminal
- 感知 preview
- 在架构上保持对 provider 中立

而对于 v1，它只需要把 Codex 做到非常出色。
