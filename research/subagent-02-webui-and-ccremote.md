# 子代理 02

范围：

- `sugyan/claude-code-webui`
- `JessyTsui/Claude-Code-Remote`

## Claude Code Web UI

### 架构

- `frontend/` 用于 React UI
- `backend/` 用于轻量级服务器/运行时包装
- 与更大的工作站型产品相比，包体占用最小
- 从截图和项目定位上可以明确看出它重视移动端 UI

### 优点

- 非常容易理解和部署。
- 对流式聊天和权限流程有很强的聚焦。
- 不依赖庞大平台栈，也有不错的移动端人体工学。
- 是一个很好的“小而锋利”的 Web 优先远程 UI 参考。

### 缺点

- 大多以 Claude 为中心。
- 相比更大型的工具，它的项目/会话/git 抽象更轻。
- 在多机器、多 agent 机群管理上能力较弱。
- 它看起来更适合“远程终端聊天”，而不是“完整远程工程控制台”。

### Panda 应该借鉴什么

- 简单的自托管 UX。
- 紧凑的流式响应循环。
- 对触控友好的审批 UI。

### Panda 应该避免什么

- 停留在一个聊天外壳上。

## Claude-Code-Remote

### 架构

- Node 守护进程
- 通道适配器：
  - 电子邮件
  - Telegram
  - LINE
- Relay/注入路径
- tmux / PTY 集成
- 用于通知驱动流程的 hooks 和监控脚本

### 优点

- 在样本集合里拥有最好的通知优先模型。
- 很适合“离席操作者”这个使用场景。
- 使用现有本地运行时，而不是强迫用户切换到新的 UI 运行时。
- 消息平台是一种现实可行且摩擦很低的远程入口。

### 缺点

- 不是一个丰富的会话浏览器。
- 与 Web UI 相比，它在 git/文件/项目 UX 上较弱。
- ChatOps 非常适合 prompt 和审批，但不适合代码检查和分支操作。
- 以 Claude 为先，没有很强的双 provider 抽象。

### Panda 应该借鉴什么

- 通知通道抽象。
- 审批升级流程。
- “回复即可继续” 的异步交互模式。
- 在主 Web UI 之外提供一个可选的旁路控制面。

### Panda 应该避免什么

- 把消息平台做成主要的操作者体验。

## 组合结论

这一对项目有价值，是因为它覆盖了两端：

- `claude-code-webui` 是偏薄的视觉外壳。
- `Claude-Code-Remote` 是偏薄的异步通知外壳。

Panda 应该同时支持这两种模式：

- 通过 PWA 进行主控制
- 通过 push 或聊天适配器接收异步提醒
